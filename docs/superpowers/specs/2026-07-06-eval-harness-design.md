# LLM Evaluation Harness — Design

**Date:** 2026-07-06
**Status:** Approved for implementation planning
**Repo slot:** Replaces the chosen idea for `agent-capstone/` (GOALS.md, Days 4-5). Renamed to `eval-harness/`.

## Context

This project prepares for a Link Ventures internal-tooling role starting 2026-07-20.
It replaces the originally-scaffolded `agent-capstone/` idea list with a concrete
choice: an **LLM Evaluation Harness** that compares Claude models on coding-adjacent
tasks. A second project from the original handoff doc, an "AI Coding Agent for
Non-Coders," was scoped during brainstorming (template-skeleton + free-form-logic +
one-retry architecture) but is **parked as a stretch/afterward item** — not required
for this capstone, may be picked up later with a lighter-weight version of that design.

## Goals

- Demonstrate AI evaluation skills: multi-model comparison, LLM-as-judge, cost/quality
  tradeoff analysis — directly applicable to evaluating tools at the studio.
- Reuse and reinforce Docker skills from `docker-101/` via sandboxed code execution.
- Produce a concrete, defensible finding (e.g. "model X beats model Y on cost at
  near-equal quality") usable in interviews/portfolio.

## Explicit scope decisions

- **Models:** Anthropic only — Opus, Sonnet, Haiku, plus prompt/temperature variants.
  No OpenAI/Gemini integration (no new API keys, no cross-vendor story for v1).
- **Task types:** two —
  1. **Code generation**, scored by running real hand-written unit tests.
  2. **API design**, scored by LLM-as-judge against a hand-written rubric.
- **Judge:** Opus acts as judge for API-design tasks and is *excluded* from being a
  scored subject on that task type, to avoid self-evaluation bias. Opus can still be
  a subject on code-gen tasks (test-scored, not judge-scored, so no bias concern there).
- **Suite size:** 5 code-gen tasks + 5 API-design tasks for the MVP — small enough to
  hand-write quality ground-truth tests/rubrics, large enough for a real comparison.
- **Execution model:** sequential runner (loop over model×task pairs, one at a time).
  No async/parallel execution — simplicity over speed for a one-off eval.
- **Sandboxing:** plain Docker (docker-py), not E2B/Modal. Directly reuses docker-101
  skills; those alternatives are compared conceptually in the write-up only.
- **Reporting:** static, self-contained HTML report + CSV export, generated by a
  script reading from SQLite. No live dashboard for v1 — **kept in mind as a future
  option** (a FastAPI+React dashboard could read the same SQLite DB later without
  changing the harness itself).
- **Storage:** SQLite, one `results` table, append-only across runs (each run tagged
  with a timestamp/run ID so historical runs remain comparable).

## Architecture & Components

1. **Task suite** (`tasks/`) — 10 task definition files (YAML/JSON): 5 code-gen
   (prompt + starter signature + hand-written pytest file), 5 API-design (prompt +
   judge rubric criteria).
2. **Model configs** (`configs.py`) — list of `ModelConfig` (model, temperature,
   prompt variant label, `judge_only: bool`).
3. **Provider client** (`llm_client.py`) — thin Anthropic SDK wrapper: given a
   `ModelConfig` + prompt, returns response text, token counts, latency, and cost
   (computed from a static pricing table). Uses the SDK's built-in retry/backoff
   (up to 3 attempts) for transient failures.
4. **Scorer** (`scoring/`) — common interface `score(task, response) -> ScoreResult`:
   - `test_scorer.py`: writes generated code to a temp dir, runs it in a short-lived
     Docker container (mounted with the task's pytest file), executes pytest under a
     hard timeout (e.g. 10s), captures pass/fail + stdout/stderr, tears the container
     down. Failures here are *not* retried — a failing test is itself the result.
   - `judge_scorer.py`: calls Opus with the response + rubric, parses a structured
     0-10 score + rationale. One retry with a stricter "JSON-only" nudge if parsing
     fails; otherwise logs `error`, `score = null`.
5. **Storage** (`storage.py`) — SQLite `results` table: `run_id, model, task_id,
   task_type, score, pass_fail, cost, latency_ms, timestamp, raw_response, error`.
   Each row is written immediately after scoring (not batched at the end), so an
   interrupted run leaves valid partial data.
6. **Runner** (`run_eval.py`) — builds the run matrix (every eligible config × every
   task of a type it can run; the judge config only invoked *as judge*, never as a
   scored subject on API-design), executes sequentially, prints one-line progress per
   run, then invokes the reporter automatically at the end.
7. **Reporter** (`report.py`) — reads the full `results` table, renders a static HTML
   file (comparison table + a cost-vs-quality chart) and a CSV export.

## Data Flow

1. Load tasks + configs.
2. Build run matrix.
3. For each `(config, task)` pair: call LLM → route to matching scorer → write result
   row to SQLite immediately → print progress line.
4. After the matrix completes, regenerate the static HTML + CSV from the full table.
5. Re-running `run_eval.py` appends a new run (new `run_id`), never overwrites.

## Error Handling

- **API failures** (rate limit/timeout/5xx): retry via SDK backoff, up to 3 attempts;
  if still failing, log the row with `error` set, `score = null`, continue the run.
- **Docker/test failures**: not retried (signal, not noise); captured as
  `pass_fail = fail` with logs in `raw_response`/`error`; hard timeout prevents a hang.
- **Judge parsing failures**: one retry with a stricter prompt; otherwise logged as
  `error`, `score = null`.
- **Whole-run resilience**: immediate per-row writes mean a crash or Ctrl-C loses at
  most the in-flight row, not the whole run.

## Testing Plan

- Unit tests for `test_scorer.py` and `judge_scorer.py` against mocked LLM responses
  (no API cost).
- One "golden" task with a known-good and a known-bad solution, used to confirm
  `test_scorer` correctly passes/fails before trusting it on real model output.
- `storage.py` tested against a temp SQLite file (write, read back, assert schema).

## Success Criteria

- All configured model×task combinations run end-to-end without crashing.
- Static HTML report renders a clear comparison table + at least one chart.
- A concrete, stated finding (e.g. cost/quality tradeoff between two models).
- Docker used correctly for sandboxed execution (timeout + teardown per run).
- README/postmortem documenting what worked, what's fragile, and what you'd harden
  for production use.
